## Summary

During the 7-sprint v1.1 implementation, several gaps were identified and resolved. This document tracks each gap, its impact, and the resolution.

## Critical Gaps (Resolved)

### Gap 1: Backchannel Handler Files Not Mounted

**Issue:** ILIAS, Moodle, and BBB backchannel handler files existed in the repository but were not mounted into running containers.

**Impact:** Backchannel logout would fail because handlers couldn't receive SAML LogoutRequests.

**Root Cause:** Volume mounts were configured for ILIAS but not replicated to Moodle and BBB, which had identical requirements.

**Resolution:**
- Added `extraVolumes` and `extraVolumeMounts` to all three services
- Created ConfigMap templates for Moodle and BBB handlers
- Pattern: Mount handler file from ConfigMap to application path

**Files Changed:**
- `helmfile/apps/ilias/values.yaml.gotmpl` (already correct)
- `helmfile/apps/moodle/values.yaml.gotmpl` (fixed)
- `helmfile/apps/bigbluebutton/values.yaml.gotmpl` (fixed)
- `helmfile/apps/moodle/templates/backchannel-handler-configmap.yaml` (new)
- `helmfile/apps/bigbluebutton/templates/saml-backchannel-controller-configmap.yaml` (new)

**Verification:**
```bash
# After deployment, verify files are mounted
kubectl exec -it moodle-pod -- ls -la /var/www/moodle/auth/shibboleth/backchannel-handler.php
kubectl exec -it bbb-pod -- ls -la /home/app/greenlight/app/controllers/saml_backchannel_controller.rb
kubectl exec -it ilias-pod -- ls -la /var/www/html/auth/shibboleth/backchannel-handler.php
```

### Gap 2: Missing Secrets Templates

**Issue:** CronJob manifests referenced Kubernetes Secrets that had no template files.

**Impact:** Deployments would fail with `Secret not found` errors. Operators wouldn't know what secrets were required.

**Root Cause:** Secrets were assumed to be created manually or via external secrets management.

**Resolution:**
- Created `secrets.yaml.template` with all required secrets
- Added comprehensive comments for credential generation
- Included warnings about not committing actual credentials
- Provided kustomize example for secure deployment

**Files Changed:**
- `scripts/user_import/kubernetes/secrets.yaml.template` (new)

**Secrets Documented:**
1. `user-provisioning-secrets` - UDM API, Keycloak API, IAM API tokens
2. `ilias-shibboleth-certs` - SAML SP certificate and private key
3. `nextcloud-oidc-secrets` - OIDC backchannel secret + client secret
4. `opencloud-oidc-secrets` - OIDC backchannel secret + client secret

**Deployment Instructions:**
```bash
# NEVER commit actual secrets - use kustomize or external secrets
kustomize edit add secret user-provisioning-secrets \
  --from-literal=udm-api-password="$(openssl rand -base64 32)" \
  --from-literal=keycloak-api-password="$(openssl rand -base64 32)"

# Or use sealed-secrets / external-secrets / Vault
```

### Gap 3: ConfigMap Placeholder Values

**Issue:** ConfigMap templates contained `example.com` placeholder values.

**Impact:** Operators might deploy with example values, causing authentication failures.

**Resolution:**
- Added prominent comments in ConfigMap template
- Documented required customization in runbook
- Values to customize:
  - `import-domain`: Your actual domain
  - `keycloak-url`: Your Keycloak URL
  - `iam-api-url`: Your IAM API endpoint

**Files Changed:**
- `scripts/user_import/kubernetes/rbac.yaml` (comments added)

### Gap 4: ILIAS Keycloak Client Missing Backchannel Logout URL

**Issue:** ILIAS SAML client `ilias-saml` had `frontchannel.logout.url` configured but no `backchannel.logout.url`, unlike all other SAML/OIDC clients (OX, Synapse, Nextcloud, OpenProject, OpenCloud).

**Impact:** Keycloak would not send SAML LogoutRequests to ILIAS during backchannel logout. ILIAS sessions would remain active after a global logout.

**Root Cause:** The SAML client was configured before the backchannel logout pattern was established. Other services (OpenCloud, Nextcloud) were added later with the pattern, but ILIAS was never retrofitted.

**Resolution:**
- Added `backchannel.logout.session.required: "true"` to ILIAS SAML client attributes
- Added `backchannel.logout.url: "https://lms.opendesk.example.com/Shibboleth.sso/Logout"` to ILIAS SAML client attributes

**Files Changed:**
- `helmfile/apps/nubus/values-opendesk-keycloak-bootstrap.yaml.gotmpl` (2 lines added)

**Verification:**
```bash
# After deployment, verify Keycloak sends LogoutRequests to ILIAS
kubectl logs -n keycloak keycloak-pod | grep "ILIAS.*logout"
```

### Consideration 4: Disaster Recovery Coverage for v1.1 Features

**Status:** Documented, needs update

**Issue:** The existing disaster recovery guide (`docs/disaster-recovery.md`) covers core service recovery (Keycloak, ILIAS, Moodle, Nextcloud, BBB) but lacks procedures for v1.1-specific features:

1. **SOPS-encrypted secrets recovery** — GnuPG/age private keys must be restored before secrets can be decrypted
2. **Backchannel logout recovery** — Keycloak SAML/OIDC client attributes must be reconfigured with correct backchannel URLs
3. **User provisioning cronjob recovery** — CronJob manifests, RBAC, and secrets must be redeployed after cluster rebuild

**Action Required:**
- Add dedicated v1.1 recovery procedures to disaster recovery guide
- Include SOPS key recovery steps
- Add backchannel logout client configuration restore steps
- Add user provisioning infrastructure redeployment steps

**Documentation:** `docs/disaster-recovery.md`

## Lessons Learned

1. **Volume mounts are easy to forget**: Create a checklist for file-based configurations
2. **Secrets need templates**: Even if using external secrets management, document what's required
3. **Test deployments early**: These gaps would have been caught in first staging deployment
4. **Consistent patterns help**: ILIAS pattern should have been immediately replicated to Moodle/BBB
5. **Cross-reference all clients for missing attributes**: ILIAS had frontchannel.logout.url but not backchannel.logout.url — always check all similar clients when adding a new attribute pattern
6. **Update DR docs alongside new features**: v1.1 added SOPS, backchannel logout, and user provisioning, but disaster-recovery.md was never updated — add DR coverage to the feature checklist
